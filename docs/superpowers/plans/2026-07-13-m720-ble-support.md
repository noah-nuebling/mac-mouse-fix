# Logitech M720 BLE Support Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在第一阶段为 Logitech M720 Triathlon 的 BLE 连接提供可逆、可测试的 HID++ 2.0 支持，把左倾、右倾、拇指键稳定映射为 Mac Mouse Fix 按钮 6、7、8，同时完整保留未配置按键的原生行为。

**Architecture:** 新增纯 Swift HID++ codec、串行请求管线、每设备 session、崩溃恢复 journal 和 BLE IOHID transport；`M720HIDPPController` 在现有 `DeviceManager` 生命周期与 `SwitchMaster` 策略之间协调。Core Graphics 与 HID++ 输入统一进入真实 `Device` 驱动的 `ButtonInputContext`。Add Mode 通过现有 message port 改成 request-ID 异步握手。所有硬件无关行为由 Helper-hosted XCTest 驱动，真实设备仅用于最后的只读诊断和验收。

**Tech Stack:** Objective-C/Swift 5、AppKit、IOKit/IOHID、CFMessagePort、XCTest、Xcode shared schemes、property-list journal。

## Global Constraints

- 规格来源是 `docs/superpowers/specs/2026-07-13-m720-ble-support-design.md`；本计划与规格冲突时以规格为准，并先修正计划再编码。
- 第一阶段只支持精确的 BLE `VID=0x046D`、`PID=0xB015`、transport=`Bluetooth Low Energy`。不得宣传或实现 Unifying/Bolt/USB receiver。
- 只管理 CID `0x005B`、`0x005D`、`0x00D0`；绝不 divert `0x0052`、`0x0053`、`0x0056`，绝不发送 feature `0x1C00`。
- 复用 `Device` 已打开的 `IOHIDDeviceRef`。M720 代码不得再次 open/close、schedule/unschedule 或 cancel 该设备。
- HID++ 处理全异步；主线程不得等待设备回复，不得嵌套 RunLoop，不得 wall-clock sleep。
- 所有定时测试使用手动 scheduler；所有外设行为使用 scripted fake transport；真实硬件测试放在最后。
- 每个任务遵循 Red → Green → Refactor：先运行精确测试看到预期失败，再写最小实现，再运行相关 suite。
- 现有 `.github/workflows` 没有 macOS XCTest runner，所以没有可优先执行的远程单测。若实现期间新增远程 runner，先运行相同 shared scheme；否则使用本文的本地 `xcodebuild`。
- 不修改现有普通 App target `Tests`。新测试 bundle 只通过 `@testable import Mac_Mouse_Fix_Helper` 测试 Helper 中同一份生产代码，生产文件不得重复编进测试 bundle。
- 每个任务只提交本任务相关文件。提交后检查 `git diff --check`；阶段结束统一 push 当前 `codex/m720-ble-support` 分支，不主动部署。

核心边界固定如下，后续任务不得绕开这些边界直接耦合 IOHID、UI 或全局状态：

```swift
protocol HIDPPTransport: AnyObject {
    var deviceIndex: UInt8 { get }
    var acceptedResponseDeviceIndices: Set<UInt8> { get }
    var onReport: ((Data) -> Void)? { get set }
    func send(_ report: Data, completion: @escaping (IOReturn) -> Void)
    func invalidate(completion: @escaping () -> Void)
}

extension HIDPPTransport {
    func invalidate() { invalidate(completion: {}) }
}

protocol HIDPPScheduler {
    var now: TimeInterval { get }
    @discardableResult
    func schedule(after delay: TimeInterval, _ block: @escaping () -> Void) -> HIDPPCancellation
}

protocol HIDPPCancellation: AnyObject {
    func cancel()
}

protocol M720ButtonEventSink: AnyObject {
    func emit(device: Device, button: Int, downNotUp: Bool)
    func cancel(device: Device, button: Int, completion: @escaping () -> Void)
}
```

`M720HIDPPSession` 和 `M720HIDPPController` 只在主线程改变生命周期状态；`HIDPPRequestPipeline` 的 transport/scheduler completion 必须 marshal 回其注入的 state queue。`Buttons` 与 `ClickCycle` 的内部状态只在 `Buttons.queue` 改变。

- [ ] **Step 2: Implement the frame and identity types**

Use these public shapes:

```swift
struct HIDPPRequestIdentity: Equatable, Hashable {
    let featureIndex: UInt8
    let function: UInt8
    let softwareID: UInt8
}

struct HIDPPErrorFrame: Equatable {
    let identity: HIDPPRequestIdentity
    let code: UInt8
}

enum HIDPPInbound: Equatable {
    case response(identity: HIDPPRequestIdentity, parameters: Data)
    case event(featureIndex: UInt8, event: UInt8, parameters: Data)
    case error(HIDPPErrorFrame)
}

enum HIDPPFrameError: Error, Equatable {
    case invalidLength(Int)
    case invalidReportID(UInt8)
    case invalidDeviceIndex(UInt8)
    case invalidSoftwareID(UInt8)

```

`request` must require function `0...15`, software ID `1...15`, parameters at most 16 bytes, pad to 20 bytes, and always use report ID `0x11`. `decode` must require exactly 20 bytes. If byte 2 is `0xFF`, decode bytes 3/4/5 as the special error layout. For normal frames, a low nibble of zero is an event; a nonzero nibble is a response.

final class ScriptedHIDPPTransport: HIDPPTransport {
    let deviceIndex: UInt8 = 0xFF
    let acceptedResponseDeviceIndices: Set<UInt8> = [0x00, 0xFF]
    var onReport: ((Data) -> Void)?
    private(set) var sent: [Data] = []

    func send(_ report: Data, completion: @escaping (IOReturn) -> Void) {
        sent.append(report)
        completion(kIOReturnSuccess)
    }

    func inject(_ bytes: [UInt8]) { onReport?(Data(bytes)) }
    func invalidate(completion: @escaping () -> Void) {
        onReport = nil
        completion()
    }
}
```

The first call immediately tells controller/Add Mode to reject new work, closes all gates, begins session cleanup, and schedules a three-second deadline. Later calls append completions. ```swift
func beginShutdown(completion: @escaping (Bool) -> Void) {
    switch state {
    case let .finished(result): completion(result)
    case .running: completions.append(completion)
    case .idle:
        state = .running
        completions = [completion]
        deadline = scheduler.schedule(after: 3.0) { [weak self] in self?.finish(false) }
        cleanup { [weak self] in self?.finish(true) }
    }
}

private func finish(_ result: Bool) {
    guard case .running = state else { return }
    state = .finished(result)
    deadline?.cancel()
    let callbacks = completions
    completions.removeAll()
    callbacks.forEach { $0(result) }
}
```
